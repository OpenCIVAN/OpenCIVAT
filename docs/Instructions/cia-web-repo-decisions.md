# Architecture Decision Records (ADRs)

This document captures key architectural decisions made for CIA Web. Understanding *why* things are the way they are helps contributors make consistent decisions.

## Format

Each decision includes:
- **Context**: What situation led to this decision
- **Decision**: What we decided
- **Consequences**: What this means for the codebase

---

## Data Architecture

### ADR-001: Three-Layer Data Model

**Context**: Scientific visualization tools often conflate data, view state, and rendering, making collaboration and state management difficult.

**Decision**: Separate into three distinct layers:
1. **Dataset** - Raw data + annotations (immutable)
2. **ViewConfiguration** - How to view data (collaborative)
3. **InstanceWindow** - GPU renderer (ephemeral)

**Consequences**:
- Closing a window doesn't lose view settings
- Multiple views of the same dataset are independent
- Clear ownership: annotations belong to Dataset, not View
- More entities to manage, but clearer responsibilities

---

### ADR-002: Server Authority for Persistent State

**Context**: Scientific research requires audit trails and reproducibility. Client-only state can be lost and can't be audited.

**Decision**: All persistent state must go through the server REST API. The server is the single source of truth.

**Consequences**:
- Every state change is logged for audit
- Offline mode requires careful sync logic
- Slightly higher latency for state changes
- Y.js is limited to ephemeral presence data only

---

### ADR-003: Server-Generated IDs for ViewConfigurations

**Context**: Need to track who created what and when for scientific reproducibility.

**Decision**: ViewConfiguration IDs are generated by the server, not the client.

**Consequences**:
- Creating a view requires an API call
- Duplicating a view creates a truly independent entity (new ID)
- Full audit trail of view creation/modification
- Can't create views while offline (must queue)

---

### ADR-004: Annotations Belong to Dataset, Not View

**Context**: Annotations are observations about the data itself, not about how you're viewing it.

**Decision**: Annotations are stored with Dataset (Layer 1), not ViewConfiguration (Layer 2).

**Consequences**:
- Annotations persist regardless of view changes
- Multiple views show the same annotations
- Views can filter which annotations to display
- Annotation visibility is a view setting, but annotation data is not

---

## Canvas & Workspace

### ADR-005: Infinite Pinboard Canvas Model

**Context**: Users need flexible workspace layouts that can grow with their analysis.

**Decision**: Canvas is a conceptually infinite 2D grid that users navigate via a viewport window.

**Consequences**:
- Only views in viewport are rendered (GPU optimization)
- Viewport snaps to whole cells (never partial views)
- Canvas dimensions grow on demand
- Need a mini-map for navigation

---

### ADR-006: View Spanning (Merge Cells)

**Context**: Some views need more space than others (like a main analysis view vs. thumbnails).

**Decision**: Placements can span 1-3 rows and 1-3 columns (like Excel merge cells).

**Consequences**:
- More flexible layouts
- Grid positioning becomes more complex
- Need to check for overlaps when placing/resizing
- Maximum 3×3 span keeps things manageable

---

### ADR-007: Three-Level Workspace Hierarchy

**Context**: Users need different collaboration scopes - sometimes private, sometimes shared.

**Decision**: Three workspace levels:
- **Personal**: Private to user
- **Breakout Room**: Shared with subset of team
- **Project Room**: Shared with all project members

**Consequences**:
- Users can explore privately before sharing
- Sub-teams can collaborate without disturbing others
- Presentations happen in Project Room
- More canvases to manage

---

### ADR-008: Subsets for Focus Mode

**Context**: Users need to focus on specific views for deep analysis without losing context.

**Decision**: Subsets are saved selections of placements that can temporarily replace the viewport.

**Consequences**:
- "Focus mode" is entering a subset
- Subsets are persisted (can be shared, revisited)
- Subsets become research artifacts with attached notes
- Need to track "exit focus" to return to previous position

---

## Plugin Architecture

### ADR-009: Handlers Provide Data, Not React Components

**Context**: Core UI should not depend on specific visualization libraries.

**Decision**: Handlers return toolbar configurations as plain JavaScript objects. The UI renders them generically.

**Consequences**:
- Core UI never imports VTK, NGL, Plotly, etc.
- New handlers don't require UI code changes
- Slightly more complex handler interface
- UI needs generic renderers for each tool type

---

### ADR-010: VTK Handler as Reference Implementation

**Context**: Contributors need a clear example to follow when adding new handlers.

**Decision**: The VTK handler in `src/core/instances/types/vtk/` serves as the canonical reference.

**Consequences**:
- VTK handler must be exceptionally well-organized
- Other handlers should copy its patterns
- VTK-specific features should not leak into the interface
- Maintaining VTK handler is high priority

---

### ADR-011: Handlers Define Their Own VR Support

**Context**: Not all visualization types make sense in VR.

**Decision**: Each handler declares `supportsVR()` and implements VR methods if applicable.

**Consequences**:
- VR support is opt-in per handler
- Graceful fallback when handler doesn't support VR
- VR configuration is type-specific
- Core VR system must handle mixed support

---

## VR Design

### ADR-012: VR-First, Desktop-First Implementation

**Context**: VR is a key differentiator, but debugging in VR is hard.

**Decision**: Design architecture for VR from day one, but implement desktop mode first.

**Consequences**:
- VR stubs exist throughout the codebase
- Components consider VR implications early
- Desktop mode is essentially "VR lite"
- Less rework when adding VR

---

### ADR-013: Curved Grid Layout in VR

**Context**: VR users shouldn't have to turn completely around to see all views.

**Decision**: Views are arranged in a curved arc around the user (like Fiesta).

**Consequences**:
- Natural viewing without 360° rotation
- Canvas positions map to arc positions
- Gaze/point selection for views
- Need careful positioning math

---

### ADR-014: Isolation Mode for Deep Analysis

**Context**: VR's killer feature is spatial exploration of 3D data.

**Decision**: Users can pull a single view to room-scale and physically walk around it.

**Consequences**:
- View scales up to room size
- Other users appear as avatars in the space
- Desktop users' cursors visible as floating pointers
- Need "return to grid" interaction

---

### ADR-015: Cross-Platform Cursor Visibility

**Context**: Collaboration requires seeing what others are pointing at, across platforms.

**Decision**: Desktop cursors visible in VR as floating dots; VR pointers visible on desktop as ray indicators.

**Consequences**:
- Handlers must project between screen and world coordinates
- Y.js presence includes cursor data
- Visual design for cross-platform indicators
- Latency considerations for cursor sync

---

## Rendering

### ADR-016: Hybrid Rendering Model

**Context**: VR headsets have limited GPUs; large datasets overwhelm browsers.

**Decision**: Support three render modes per handler:
- **Client**: Browser GPU via WebGL
- **Server**: Remote GPU, video streamed
- **Hybrid**: Mix based on complexity

**Consequences**:
- Handlers must implement mode switching
- Need server-side rendering infrastructure
- WebRTC for video streaming
- More complex but more scalable

---

### ADR-017: Target <32ms Latency for VR

**Context**: High latency in VR causes motion sickness.

**Decision**: Interaction-to-photon latency must be under 32ms for VR mode.

**Consequences**:
- Hardware video encoding (NVENC)
- Regional render servers
- May sacrifice quality for latency
- Predictive rendering techniques

---

## Collaboration

### ADR-018: Y.js for Presence Only

**Context**: Y.js is great for CRDTs, but we need server authority for audit trails.

**Decision**: Y.js handles only ephemeral presence data (cursors, avatars, online status). All persistent state goes through REST API.

**Consequences**:
- Clear separation of concerns
- Server remains source of truth
- Y.js reconnection doesn't cause state conflicts
- Need to handle presence vs. state differently in UI

---

### ADR-019: Link Modes for View Synchronization

**Context**: Users sometimes want to follow along, sometimes explore independently.

**Decision**: Three link modes:
- **Follow**: One-way sync (I follow you)
- **Bidirectional**: Two-way sync (we move together)
- **Broadcast**: Source controls all (presentation mode)

**Consequences**:
- Fine-grained collaboration control
- Complex link management logic
- Need to handle broken links gracefully
- UI must clearly indicate link status

---

## Technical Conventions

### ADR-020: Webpack Aliases for Imports

**Context**: Deep relative imports (`../../../`) are hard to maintain.

**Decision**: Use webpack aliases: `@Core`, `@UI`, `@Collaboration`.

**Consequences**:
- Clean, consistent imports
- Easy to refactor file locations
- Need to configure in webpack and IDE
- All contributors must use aliases

---

### ADR-021: BEM Naming for CSS

**Context**: CSS naming conflicts and specificity issues.

**Decision**: Use BEM naming: `.block__element--modifier`.

**Consequences**:
- Predictable, flat CSS
- No nested selector specificity issues
- Verbose class names
- Clear component boundaries

---

### ADR-022: Component Logic Separation

**Context**: Designers and developers often work on the same components.

**Decision**: Separate logic (`.logic.js`) from presentation (`.jsx`).

**Consequences**:
- Designers can modify UI without touching logic
- Logic is more testable
- Additional files per component
- Clear interface between logic and presentation

# Visualization State Layers

CIA Web / OpenCIVAN uses a strict three-layer architecture to separate raw scientific data from visualization configuration and from rendering runtime state. Keeping these layers distinct is what makes collaborative views reproducible, shareable, and reconstructable on any compatible client — including Apple Vision Pro thin clients receiving server-rendered frames.

---

## 1. Three-Layer Architecture Overview

| Layer | Class / Location | Responsibility | Persisted |
|-------|-----------------|----------------|-----------|
| **Layer 1 — Dataset** | `Dataset.js` / `DatasetManager.js` | Raw bytes + identity metadata | PostgreSQL + MinIO |
| **Layer 2 — ViewConfiguration** | `ViewConfiguration.js` / `ViewConfigurationManager.js` | Declarative visualization specification | PostgreSQL + REST |
| **Layer 3 — InstanceWindow / Render Instance** | `VTKInstanceHandler.js` | Ephemeral VTK.js runtime pipeline | Never — per-client only |

The dependency direction is one-way: Layer 3 is built from Layer 1 and Layer 2. Nothing in Layer 1 or Layer 2 should reference Layer 3 objects.

---

## 2. What Each Layer Stores — and Must Not Store

### Layer 1: Dataset

**Stores:**
- Stable `id` (server-generated UUID)
- `hash` — SHA-256 of file bytes (deduplication + identity verification)
- `storageKey` — MinIO object path
- `publicPath` — relative URL for built-in samples (e.g. `/vtp_files/Bones.vtp`)
- `fileType` — file format extension
- `metadata.fileSize` — byte count
- `metadata.isBuiltIn` — whether the file ships with the platform
- `data_arrays` — derived scalar array names (JSONB on server)

**Must not store:**
- Rendered frames or screenshots as durable content
- ViewConfiguration state (camera, colormap, filters)
- User presence or session state
- VTK.js runtime objects

### Layer 2: ViewConfiguration

**Stores (as of DR2):**
- `id`, `name`, `description` — view identity
- `schemaVersion` — integer; bump when adding non-backward-compatible fields
- `datasetRefs[]` — array of `{ datasetId, contentHash, format, sizeBytes, role }` — Layer 1 identity contracts
- `datasetId` — legacy single ref; kept for backward compat; migrated to `datasetRefs` automatically
- `camera` — camera state object (position, focal point, view-up)
- `filters[]`, `widgets[]`, `colorMaps` — visualization parameters
- `time` — time-series playback state (see §6)
- `composition[]` — multi-dataset layer ordering (see §7)
- `compatibility` — declarative render requirements (see §5)
- `renderMode` — `'local' | 'remote' | 'auto'`
- `snapshots[]` — named restore points
- `links` — selective property linking between views
- `broadcast`, `following` — live presentation state
- `vrHints` — VR scale/origin preferences

**Must not store:**
- Raw dataset bytes or MinIO credentials
- Browser `blob:` or `data:` URLs as durable identifiers
- Live VTK.js objects: actors, mappers, renderers, pipelines
- WebGL handles, GPU buffers, or OffscreenCanvas references
- DOM element references
- Y.js Y.Map / Y.Array references
- Keycloak access tokens

### Layer 3: InstanceWindow / Render Instance

**Stores (ephemeral, per-client):**
- VTK.js renderer, renderWindow, interactor
- VTK actors, mappers, filters pipeline
- WebGL context and GPU buffers
- Parsed VTK data objects (polydata, image data, etc.)
- Animation timers and playback intervals

**Layer 3 must be recreated from Layer 1 + Layer 2** every time a client mounts a view. It must not be serialized or transmitted over the network.

## 3. Dataset Hash-Based Reproducibility

Every uploaded dataset gets a SHA-256 hash computed from its bytes, stored in the `datasets.hash` column. This hash is the dataset's content identity — two datasets with the same hash are byte-for-byte identical regardless of filename.

When a ViewConfiguration references a dataset, it stores `datasetRefs[].contentHash`. On reconstruction, the client can compare this expected hash against the local `Dataset.hash` to confirm it is rendering the exact expected data.

**Hash status values** (`DatasetIdentityService.DATASET_STATUS`):

| Status | Meaning |
|--------|---------|
| `verified` | `contentHash` matches `Dataset.hash` — byte-identical |
| `unverified` | No hash in ref or local dataset — cannot confirm |
| `missing` | Dataset not registered in local manager |
| `hash_mismatch` | Hash present but does not match — data may differ |
| `unsupported` | Format not supported on this client |

**Built-in datasets** have SHA-256 hashes in `public/vtp_files/manifest.json`, generated by `scripts/hashBuiltinManifest.js`. When a client loads a built-in dataset, the hash in the manifest is compared against the loaded bytes to confirm byte-level identity.

### Regenerating built-in dataset hashes

Run after adding or replacing any `.vtp` file in `public/vtp_files/`:

```bash
npm run hash:builtin                         # recompute and write manifest.json
npm run build:dataset-manifest -- --validate # check without writing (used by npm run build)
```

Commit the updated `manifest.json` alongside the changed `.vtp` file. The `npm run build` prebuild step validates hashes automatically — it will fail if committed hashes are stale, catching cases where a VTP file was replaced without regenerating the manifest.

## 4. ViewConfiguration Schema Versioning and Migration

`ViewConfigurationSchema.js` exports:

- `SCHEMA_VERSION` — current integer (starts at 1)
- `normalizeViewConfiguration(raw)` — pure function; fills missing DR2 fields with safe defaults; migrates legacy `datasetId` → `datasetRefs[]`
- `validateViewConfiguration(config)` — pure function; returns `{ valid, errors, warnings }`
- `migrateDatasetIdToRef(datasetId)` — builds a minimal `datasetRef` entry from a bare id

### Migration rules

- Old configs with no `schemaVersion` → treated as version 1 (always compatible)
- Old configs with `datasetId` but no `datasetRefs` → `datasetRefs` is synthesized with `contentHash: null`
- Unknown extra fields in the config object → passed through unchanged (forward-compatible reads)
- Invalid critical fields (missing `id`, missing `datasetId` in a ref entry) → `validateViewConfiguration` returns errors

### When to bump SCHEMA_VERSION

Bump only when removing or renaming a field that older clients actively rely on. Adding new optional fields is always backward-compatible and does not require a version bump. When bumping, add a migration case to `normalizeViewConfiguration`.

## 5. Rendering Compatibility Checks

`ClientCapabilityProfile.js` provides:

- `buildClientCapabilityProfile([overrides])` — detects WebGL/WebGL2/WebXR availability via feature detection (not UA string)
- `compareRenderCapabilities(viewConfig, datasetResolution, clientProfile)` — returns a `CONSISTENCY_STATUS` string
- `CONSISTENCY_STATUS` enum

**Consistency status values:**

| Status | Meaning |
|--------|---------|
| `consistent` | Dataset hash verified + all required features supported |
| `compatible_unverified` | No hash to verify, but features are compatible |
| `degraded` | Optional feature (e.g. WebXR) unavailable; view loads with reduced fidelity |
| `incompatible` | Critical mismatch: missing dataset, hash mismatch, or unsupported format |
| `unknown` | Insufficient information (no dataset refs, or profile unavailable) |

**UI:** `RenderConsistencyBadge` runs these checks on mount and displays a color-coded chip. It is non-blocking — incompatible status shows a red chip but does not prevent exploration. Only a `hash_mismatch` or `missing` dataset should prompt a user-facing warning before acting on results.

## 6. Time-Series Playback State

`VTKTimeSeriesFeature` (Layer 3) manages frame-level playback state: current step, timer interval, animation loop. This state is inherently ephemeral.

**DR2 adds `viewConfig.time`** to persist the durable *analytical position* in a time series across sessions and client joins:

```js
viewConfig.time = {
  enabled:      true,
  currentStep:  5,
  totalSteps:   100,      // null = determined by dataset at load time
  playbackMode: 'paused', // 'stopped' | 'playing' | 'paused'
  direction:    1,        // 1 = forward, -1 = reverse
  loop:         'loop',   // 'once' | 'loop' | 'pingPong'
  fps:          10,
  interpolation: 'none',  // 'none' | 'linear'
  authorUserId: null,     // null = no shared playback authority
  committedAt:  null,     // timestamp of last durable save
}
```

**Rules:**
- `viewConfig.time` is written at explicit commit points (play/pause/step), not on every frame tick.
- Per-frame ticker state (animation frame IDs, interval handles) remains entirely in `VTKTimeSeriesFeature` (Layer 3).
- Late joiners read `viewConfig.time.currentStep` to seek to the shared analytical position.
- `viewConfig.time = null` means the current dataset has no time dimension or time state is not yet determined.

The wiring from `viewConfig.time` → `VTKTimeSeriesFeature.initialize()` is follow-up work (DR2.5). The schema is defined here so that ViewConfigurations saved in DR2 are forward-compatible with that wiring.

## 7. Multi-Dataset Composition

**DR2 adds `viewConfig.composition[]`** for views that layer multiple datasets:

```js
viewConfig.composition = [
  {
    datasetId:       'ds-uuid-primary',
    role:            'primary',   // 'primary' | 'overlay' | 'reference'
    visible:         true,
    opacity:         1.0,
    order:           0,           // z-order (lower = rendered first)
    scalarArrayName: null,
    colorMapId:      null,
    filters:         [],
    transform:       null,        // { translate, rotate, scale } registration
    label:           null,
  },
  {
    datasetId:       'ds-uuid-overlay',
    role:            'overlay',
    visible:         true,
    opacity:         0.6,
    order:           1,
    scalarArrayName: 'Pressure',
    colorMapId:      'cool-warm',
    filters:         [],
    transform:       null,
    label:           'Pressure field',
  },
]
```

**Rules:**
- Every entry references a Layer 1 dataset by `datasetId`.
- Runtime VTK actors/pipelines for each entry remain Layer 3 only.
- A missing overlay dataset produces `degraded` status (view still useful).
- A missing primary dataset produces `incompatible` status.
- `composition` and `datasetRefs` are maintained in sync: every `composition[].datasetId` should appear in `datasetRefs`.

VTKInstanceHandler rendering from `composition` is future DR3 work. The schema is defined now so saved ViewConfigurations are forward-compatible.

## 8. Apple Vision Pro / Thin-Client / Server-Rendered Mode

Apple Vision Pro is a primary thin-client target. The architecture is designed so that:

- **Layer 1** dataset identity still matters on the server/render host that does the actual VTK processing.
- **Layer 2** ViewConfiguration still defines the analytical view (camera, scalar selection, colormap, etc.) and is sent to both the server and the client.
- **Layer 3** rendering may happen on the Windows NVIDIA GPU server. The Vision Pro client receives rendered frames via `RemoteRenderClient.js` (WebSocket to Python VTK server), not local VTK.js pipelines.

`viewConfig.renderMode = 'remote'` marks a view as intended for server-side rendering. `ClientCapabilityProfile.buildClientCapabilityProfile({ renderMode: 'remote' })` produces a profile that reflects this.

`compareRenderCapabilities` with `renderMode: 'remote'` classifies against the server renderer's capabilities, not the local VTK.js pipeline. A view that `requiresLocalRender: true` is `degraded` on a remote-mode client, not `incompatible`.

**Do not force Vision Pro clients to download and parse large VTP files** if the intended path is server-side rendering. The client receives rendered frames; it only needs the ViewConfiguration and Layer 1 dataset identity (not the dataset bytes).

## 9. Contributor Guide: Adding a New Visualization Option

### Checklist

1. **Decide which layer it belongs to:**
   - User preference / visualization parameter? → Layer 2 (`ViewConfiguration`)
   - Raw data property? → Layer 1 (`Dataset`)
   - Runtime rendering state? → Layer 3 (`VTKInstanceHandler` / feature module)

2. **If Layer 2 (most visualization options are):**
   - Add the field to `ViewConfiguration` constructor with a safe default.
   - Add it to `ViewConfiguration.toJSON()`.
   - `fromJSON()` spreads all fields automatically — no change needed unless you need special deserialization.
   - Add normalization (default value for missing field) to `normalizeViewConfiguration()` in `ViewConfigurationSchema.js`.
   - If the field is critical for reconstruction, add validation to `validateViewConfiguration()`.
   - Add the field to `ViewSnapshot.fromView()` and `snapshot.state` if it should be saved in snapshots.
   - Add it to `fork()` if the forked view should copy it.

3. **If Layer 3 (runtime-only):**
   - Add a new VTK feature module in `src/core/instances/types/vtk/features/`.
   - Register it in the manifest (`manifest.ts`) if it requires a declared capability.
   - Never serialize VTK objects into ViewConfiguration.
   - If the feature has durable *configuration* (not runtime state), put that configuration in Layer 2 and reconstruct the runtime from it.

4. **Never:**
   - Store a `blob:` URL, DOM node, canvas reference, VTK actor, WebGL handle, or GPU buffer in `ViewConfiguration`.
   - Read `ViewConfiguration` fields from inside a VTK feature to mutate VTK pipeline state directly — the reconstruction flow should go through the `VTKInstanceHandler` public API.
   - Add a rendering option to `Dataset` — the Dataset is content identity, not visualization state.

5. **Test:**
   - Add a `normalizeViewConfiguration` test that confirms the new field defaults safely on old configs.
   - Add a `validateViewConfiguration` test if the field is critical.
   - Add a round-trip test if the field has complex serialization.
